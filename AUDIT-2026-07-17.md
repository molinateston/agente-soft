## CRÍTICOS

### C1 · bootstrap silencioso se o GitHub cair
- **Arquivo:** README.md:17 · bootstrap.sh (invocação externa)
- **Padrão:** (b) COMANDO QUE FALHA SILENCIOSO
- **Cenário:** cliente cola `sudo bash -c "$(curl -fsSL https://raw.githubusercontent.com/.../bootstrap.sh)"`. Se o GitHub estiver fora (raro, mas acontece: 500/503/DNS/rede lenta), o `curl -fsSL` retorna vazio (`-f` faz retornar não-zero, mas o `$(...)` PROPAGA a string vazia, e `sudo bash -c ""` executa nada e sai 0). Terminal fica limpo, sem erro visível, sem instalação. O cliente pergunta "e agora?" e o Léo perde 15 min diagnosticando.
- **Fix proposto:** trocar por invocação de duas linhas ou embutir um sentinel de fim no bootstrap. Padrão robusto:
  ```bash
  curl -fsSL https://raw.githubusercontent.com/molinateston/agente-soft/main/bootstrap.sh -o /tmp/bs.sh \
    && sudo bash /tmp/bs.sh \
    || { echo "✗ Não consegui baixar o instalador. Cheque sua internet e tente de novo."; }
  ```
  Alternativa mais enxuta: adicionar `set -e` no início do wrapper e um marker de fim no `bootstrap.sh` (ex: `echo "✅ BOOTSTRAP-END"`), com o comando de instalação checando por esse marker.
- **Mensagem pro dono (proposta, passou no lint):**
  > *"Não consegui baixar o instalador. Confere sua internet e roda o comando de novo. Se continuar, o GitHub pode estar fora do ar."*
- **Severidade:** CRÍTICO.

### C2 · `.env` grava valores VAZIOS se o Claude perde variáveis entre Bash calls
- **Arquivo:** SETUP-AGENTE.md:126-135 (heredoc `cat > ~/lean-bridge/.env <<EOF`)
- **Padrão:** (a) ASSUME MAS NÃO CHECA + (b) FALHA SILENCIOSA
- **Cenário:** o heredoc depende de `$TELEGRAM_BOT_TOKEN`, `$AGENT_NAME`, `$OWNER_NAME` estarem no ambiente do Bash. **Cada Bash call do Claude Code é um shell novo — env de uma call NÃO persiste automaticamente na próxima.** O manual não instrui o Claude a `export` as vars nem a rodar as ETAPAS 1-3 na MESMA call de Bash. Na prática, o Claude sabe substituir literal ao escrever, mas em cliente novo — sonnet meio distraído, ou execução parcialmente delegada — pode gravar `TELEGRAM_BOT_TOKEN=` (vazio) e seguir. A `ETAPA 4` sobe o serviço, o bridge cai no guard `if (!TG_TOKEN) process.exit(1)` (bridge.cjs:137), serviço fica em auto-restart infinito e o dono só vê "não subiu".
- **Fix proposto:** substituir o heredoc por um bloco explícito que:
  1. Checa se as 3 vars estão preenchidas ANTES do `cat >`, e ABORTA com mensagem clara se qualquer uma estiver vazia.
  2. Instrui o Claude, em nota, a substituir os placeholders **literalmente** ao escrever o `.env` (não confiar em expansão de shell).
  Ex:
  ```bash
  # Antes de gravar: garanta que os 3 estão preenchidos (senão o bridge aborta e o dono vê "não subiu")
  [ -n "$TELEGRAM_BOT_TOKEN" ] && [ -n "$AGENT_NAME" ] && [ -n "$OWNER_NAME" ] \
    || { echo "✗ Faltou token, nome do agente ou nome do dono — refaça a ETAPA 1."; exit 1; }
  ```
  E, na nota do `.env`: *"Claude: se estiver escrevendo este arquivo, cole os VALORES literais das perguntas da ETAPA 1 no lugar dos `$VAR`. NÃO deixe placeholders."*
- **Mensagem pro dono (proposta):**
  > *"Falta o token do bot ou o seu ID no arquivo de configuração. Não consigo subir sem essas duas peças. Refaz o passo do BotFather e da captura de ID."*
- **Severidade:** CRÍTICO.

### C3 · Guard de env sobe pro log, não pro dono
- **Arquivo:** bridge.cjs:137-138
- **Padrão:** (f) MENSAGEM PRA DONO EM JARGÃO (ausente, na verdade) + (e) RECUPERAÇÃO AUSENTE
- **Cenário:** consequência direta do C2 (e de qualquer edição na mão do `.env`). O bridge faz `console.error("falta TELEGRAM_BOT_TOKEN") ; process.exit(1)`. Isso vai pro `~/lean-bridge/bridge.log`. O dono está no Telegram, não olhando log. O systemd reinicia infinito (Restart=always), o healthcheck detecta serviço "active" a maior parte do tempo (porque restart-loop deixa curtos active), e o dono só percebe quando bate no `StartLimitBurst=5/300s` — 5 min depois.
- **Fix proposto:** antes do `process.exit(1)`, tentar enviar 1 mensagem direta pro `OWNER_CHAT_ID` (se ele existir) via curl-like — se OWNER também não existir, escrever num sentinel `~/lean-bridge/.startup-failed` que o healthcheck lê e traduz na próxima passagem. Bônus: o próprio healthcheck já pode detectar restart-loop (várias falhas em janela curta) e alertar.
- **Mensagem pro dono (mesma do C2):**
  > *"Falta o token do bot ou o seu ID no arquivo de configuração. Não consigo subir sem essas duas peças. Refaz o passo do BotFather e da captura de ID."*
- **Severidade:** CRÍTICO.

## ALTOS

### A1 · Update aplicado meio-a-meio quando bridge.cjs novo tem erro de sintaxe
- **Arquivo:** update.sh:164-167
- **Padrão:** (b) FALHA SILENCIOSA + (e) RECUPERAÇÃO AUSENTE
- **Cenário:** dono publica bridge.cjs novo com typo. Timer semanal do cliente puxa. `node --check` falha → `say "✗ bridge.cjs novo tem erro de sintaxe — mantido o antigo."`. NÃO seta `FAIL=1`. Skills novas foram aplicadas (git pull rodou antes na linha 101). Restart usa bridge antigo + skills novas. Se a skill nova depende de feature nova do bridge, cliente quebra em silêncio. O log tem a linha, mas nada chega no Telegram — o `trap on_exit` já foi desarmado na linha 221.
- **Fix proposto:**
  1. Setar `BRIDGE_BROKEN=1` no ramo do `else`.
  2. No final, se `BRIDGE_BROKEN=1`: mandar mensagem `tg` antes do restart, ou reverter as skills junto (rollback).
  3. Mínimo aceitável: mandar aviso via `tg`.
- **Mensagem pro dono:**
  > *"A atualização veio com defeito. Deixei tudo como estava antes, você segue no ar. Vou tentar de novo no automático."*
- **Severidade:** ALTO.

### A2 · Healthcheck é MUDO se `.env` corrompeu
- **Arquivo:** healthcheck.sh:25-28
- **Padrão:** (b) FALHA SILENCIOSA
- **Cenário:** o dono edita `.env` na mão (dizemos "libera o grupo, coloca ID no ALLOWED_SENDERS"), erra a quebra de linha ou apaga o `OWNER_CHAT_ID`. Serviço cai no guard C3. Healthcheck roda; `env_get OWNER_CHAT_ID` volta vazio; `alert()` faz `[ -n "$OWNER" ]` → falso → `|| true` → NADA. O dono não recebe aviso e acha que está tudo bem.
- **Fix proposto:** se `OWNER` vazio, `healthcheck.sh` grava sentinel `~/lean-bridge/.owner-lost` e loga alto no `health.log`; próxima instância que conseguir ler o `.env` (após rollback ou fix manual) manda um aviso de recuperação. Ou: fallback pra ler `OWNER_CHAT_ID` do último backup em `~/lean-bridge/backups/`.
- **Mensagem pro dono:** N/A (o próprio problema é a incapacidade de mandar mensagem). Aviso pós-recuperação:
  > *"Meu arquivo de configuração tinha bagunçado e eu fiquei mudo. Já voltei."*
- **Severidade:** ALTO.

### A3 · Auto-update sem `set -e` deixa passos silenciosos escaparem
- **Arquivo:** update.sh:16 (`set -uo pipefail`, sem `-e`)
- **Padrão:** (b) FALHA SILENCIOSA
- **Cenário:** decisão deliberada pra controlar `FAIL` manualmente, mas alguns `cp -p ... || true` e um monte de linhas sem `|| exit` deixam o script seguir mesmo após falha real (ex: `cp -p "$HOME/.config/systemd/user/agente.service" "$BACKUP_DIR/agente.service" 2>/dev/null || true` na linha 155 — se o snapshot do service falhar, o rollback futuro não terá o service pra restaurar).
- **Fix proposto:** manter `set -uo pipefail`, mas trocar os `|| true` de PASSOS CRÍTICOS por `|| { say "..."; FAIL=1; }`. Rodar shellcheck no arquivo.
- **Mensagem pro dono:** só em caso de falha real, já coberta por A1.
- **Severidade:** ALTO.

### A4 · Bootstrap não valida que o `agente` consegue rodar `claude`
- **Arquivo:** bootstrap.sh:69-77, 137
- **Padrão:** (a) ASSUME MAS NÃO CHECA
- **Cenário:** `npm install -g @anthropic-ai/claude-code` rodado como root instala em `/usr/local/lib/node_modules` com symlink em `/usr/local/bin/claude`. **Checa se `claude --version` roda como root** (linha 73). NÃO checa se roda como `agente`. Em VPS com `PATH` do usuário `agente` mutilado (config estranha, Ubuntu customizado), o `exec sudo -iu agente claude` na linha 137 falha com "command not found". `|| echo "..."` só imprime, mas o cliente já perdeu o terminal (`exec` substitui o processo).
- **Fix proposto:** antes do `exec`, testar `sudo -iu agente which claude` e abortar com mensagem em português se não achar, sugerindo `ln -s $(which claude) /usr/local/bin/claude` como fix.
- **Mensagem pro dono:**
  > *"O Claude instalou mas não abre pro usuário do agente. Roda esse comando pra ligar os dois: `sudo ln -s $(which claude) /usr/local/bin/claude`, depois refaz o instalador."*
- **Severidade:** ALTO.

### A5 · Captura de OWNER via `sed -i` quebra se ID vier estranho
- **Arquivo:** SETUP-AGENTE.md:197-199
- **Padrão:** (a) ASSUME MAS NÃO CHECA
- **Cenário:** `sed -i "s/^OWNER_CHAT_ID=.*/OWNER_CHAT_ID=$OWNER_CHAT_ID/"`. Se o Claude — por bug de parsing do JSON do `getUpdates` — capturar algo com `/`, `&`, aspa ou barra invertida, o `sed` quebra ou substitui errado. Chance baixa (IDs do Telegram são só dígitos, com sinal negativo em grupos), mas se o Claude confundir `chat_id` com `username` (edge case: msg de canal encaminhada), aí sim.
- **Fix proposto:** validar `[[ "$OWNER_CHAT_ID" =~ ^-?[0-9]+$ ]]` antes do `sed`. Se falhar, mostrar "Não consegui reconhecer o ID capturado. Refaz mandando a msg no bot de novo."
- **Mensagem pro dono:**
  > *"Não consegui reconhecer o seu ID direito. Manda a mesma mensagem no bot de novo pra eu tentar capturar limpo."*
- **Severidade:** ALTO.

### A6 · SETUP ETAPA 5: "serviço ATIVO mas mudo" cai em copy genérica
- **Arquivo:** SETUP-AGENTE.md:283-296
- **Padrão:** (f) MENSAGEM PRA DONO EM JARGÃO
- **Cenário:** serviço sobe, saudação não chega. SETUP faz curl direto pro Telegram. Se voltar `ENTREGA FALHOU`, o Claude repassa "revise a ETAPA 3.5" — sem dizer o que exatamente checar. Cliente novo trava.
- **Fix proposto:** o ramo de `ENTREGA FALHOU` já sabe o `error_code` do Telegram. Traduzir os 3 casos comuns: `400 chat not found` → ID errado. `401 Unauthorized` → token errado. `403 bot was blocked` → dono bloqueou o bot. Cada um com 1 linha de instrução clara.
- **Mensagens pro dono (por caso):**
  - Token errado: *"O token do bot não é válido. Copia de novo lá do BotFather e me manda."*
  - ID errado: *"O ID que eu peguei não abre conversa com você. Volta no bot e me manda a msg de captura de novo."*
  - Bot bloqueado: *"Você bloqueou meu bot no Telegram. Desbloqueia e manda 'oi' que eu volto."*
- **Severidade:** ALTO.

### A7 · Bootstrap não roda como quem deveria depois — nunca valida no user certo
- **Arquivo:** bootstrap.sh:87 (`loginctl enable-linger agente`)
- **Padrão:** (a) ASSUME MAS NÃO CHECA
- **Cenário:** `loginctl enable-linger agente` pode falhar silenciosamente numa VPS minimizada onde `systemd-logind` não está ativo (ex: containers). O `|| true` engole. Depois, o SETUP ETAPA 4 tenta `systemctl --user enable --now agente` e recebe "Failed to connect to bus". Cliente vê erro cru.
- **Fix proposto:** após o `enable-linger`, validar com `loginctl show-user agente -p Linger` e abortar cedo com mensagem clara se `Linger=no`. Ou tentar `systemctl start systemd-logind` antes.
- **Mensagem pro dono:**
  > *"Sua VPS não tem o pedaço do sistema que segura o agente ligado depois que você fecha o terminal. Provavelmente é uma VPS minimizada. Fala com o suporte pra ativar o systemd-logind, ou usa uma imagem Ubuntu padrão."*
- **Severidade:** ALTO.

### A8 · `agente.service` congela `NODE_BIN` no momento da instalação
- **Arquivo:** SETUP-AGENTE.md:233, 251
- **Padrão:** (a) ASSUME MAS NÃO CHECA (a longo prazo)
- **Cenário:** SETUP grava `NODE_BIN="$(command -v node || echo /usr/bin/node)"` e interpola no `ExecStart` do service. Se depois o cliente atualiza node (via nvm, ou o próprio `apt-get upgrade` move), o path velho pode ficar quebrado. Bridge não sobe. `bootstrap.sh` já cria symlink em `/usr/local/bin/node` se preciso, mas apenas se `command -v node != /usr/bin/node` no boot inicial — não protege mudança futura.
- **Fix proposto:** sempre gravar `ExecStart=/usr/local/bin/node ...` E garantir no bootstrap que o symlink existe INcondicionalmente (mesmo se `command -v node` já é `/usr/bin/node`, criar `/usr/local/bin/node -> /usr/bin/node` como âncora estável).
- **Mensagem pro dono:** N/A (invisível até quebrar).
- **Severidade:** ALTO.

## MÉDIOS

### M1 · `enable-voice.sh`: venv meio-instalado se pip cai
- **Arquivo:** enable-voice.sh:41-43
- **Padrão:** (e) RECUPERAÇÃO AUSENTE
- **Cenário:** `python3 -m venv` roda ok. `pip install faster-whisper` cai no meio (rede). `exit 1`, mas o `$VENV` fica com uma instalação parcial. Próxima rodada, o fast-path idempotente (linha 28-38) tenta `import faster_whisper` — pode dar `ImportError` e cair no ramo lento, refazendo do zero (OK), mas se der outro erro (ex: importa mas classe faltando), a idempotência pula e nada funciona.
- **Fix proposto:** em caso de falha do pip, `rm -rf "$VENV"` antes de sair.

### M2 · `SETUP ETAPA 3.5c` pede "sou eu, $OWNER_NAME" mas não confere o texto
- **Arquivo:** SETUP-AGENTE.md:186-192
- **Padrão:** (a) ASSUME MAS NÃO CHECA
- **Cenário:** a doc diz "A prova de identidade vem do TEXTO da mensagem". Mas o passo `d` (linha 182) mostra TODOS os remetentes e deixa a checagem pro Claude olhar no olho. Se o dono digitar `sou eu leo` (sem vírgula) ou `sou eu, Leo` (maiúscula diferente), o Claude pode aceitar ou rejeitar por critério inconsistente. Se aparecer outra pessoa que também digitou algo parecido, risco de gravar ID errado — que **controla a VPS inteira**.
- **Fix proposto:** filtrar por regex no próprio node do passo `d`: `if (!/^sou eu[, ]+.*/i.test((m.text||"")))`. Só listar quem casou. Se 0, pede pro dono repetir. Se 1, grava. Se 2+, pergunta explícito.

### M3 · `update.sh` linha 155: cp do `agente.service` no snapshot com `|| true`
- **Arquivo:** update.sh:155
- **Padrão:** (e) RECUPERAÇÃO AUSENTE
- **Cenário:** snapshot do `.service` falha silenciosamente. Update segue. Se der ruim, rollback tenta restaurar service e falha (falta arquivo no backup). Cliente fica sem service — ou com service divergente do estado atual.
- **Fix proposto:** trocar `|| true` por `|| snap_fail "agente.service"`. Trata como qualquer outro arquivo crítico.

### M4 · Healthcheck de login expirado é CIRCULAR
- **Arquivo:** healthcheck.sh:92

  > *"O meu acesso ao Claude venceu. Preciso que você entre de novo pra eu voltar a responder."*
  (passou no lint, sem jargão)

### M5 · Bootstrap.sh: `apt-get install nodejs` sem `-y --no-install-recommends`
- **Arquivo:** bootstrap.sh:55
- **Padrão:** (b) baixo risco
- **Cenário:** apt puxa "recommends" que podem falhar em rede lenta. Levar minutos a mais e às vezes cair.
- **Fix proposto:** adicionar `--no-install-recommends` em todos os `apt-get install`.

### M6 · `AGENT_NAME`/`OWNER_NAME` com espaço não é validado como "só letras+espaço+hífen"
- **Arquivo:** SETUP-AGENTE.md:56 (nota) — a validação é PROSA, não código
- **Padrão:** (a) ASSUME MAS NÃO CHECA
- **Cenário:** a nota pede pro Claude normalizar. Se ele esquece (ou cliente digita com aspa dupla), quebra o `.env` (fecha string cedo) OU o `sh -c` da saudação do boot.
- **Fix proposto:** um regex de validação em Bash antes do `.env`: `[[ "$AGENT_NAME" =~ ^[A-Za-zÀ-ÿ\ \-]+$ ]] || { echo "✗ Nome inválido"; exit 1; }`.
